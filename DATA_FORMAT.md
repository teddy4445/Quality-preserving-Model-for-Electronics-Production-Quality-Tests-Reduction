# Data Format Guide

This document describes the input data schema expected by the notebooks in this repository and the intermediate artifacts they generate.

## 1. Input Data Scope
The project expects **step-level production test logs** for printed circuit board assembly (PCBA) runs.  
Each row should represent one executed test step for one board run (unit).

You will typically prepare separate datasets for:
- FCT (Functional Circuit Test)
- EOL (End-of-Line test)
- Optional temporal validation slices for each stage

## 2. Required Columns
The notebooks rely on the following columns (case-insensitive before normalization, but names must map to these logical fields):

- `source_file`
  - String identifier of a board run / unit instance.
  - Used to build `unit_id`.
- `outcome_from_filename`
  - Unit-level outcome label.
  - Expected values: `PASS`, `FAIL`, `ABORT` (or values mapped to these).
- `result`
  - Step-level outcome label.
  - Expected values: `PASS`, `FAIL`, `ABORT`.
- `step_name_1`
  - Primary test-step category/name.
- `step_name_2`
  - Secondary/sub-step name.
- `step_time`
  - Numeric execution time per step (seconds).
- `no`
  - Record ordering/index field; used in cleaning/filtering.

## 3. Derived Fields Used by the Pipeline
Notebooks derive these fields internally:

- `unit_id = source_file`
- `test_id = step_name_1 + " / " + step_name_2`
- `result_binary`:
  - `PASS -> 1`
  - `FAIL -> 0`
  - `ABORT -> 0`
- `unit_pass`:
  - `PASS -> 1`
  - `FAIL -> 0`
  - `ABORT -> 0`

## 4. Expected Data Quality Rules
For stable execution:

- Ensure `step_time` is numeric or coercible to numeric.
- Ensure `source_file`, `step_name_1`, and `step_name_2` are not empty for valid rows.
- Keep row order for temporal-validation datasets (do not shuffle).
- Preserve all FAIL/ABORT rows; they drive coverage and escape-risk estimates.

## 5. Notebook Input Files
### 5.1 Raw/entry-point files
These are referenced by path variables inside notebooks and should be updated locally:

- FCT raw file path variable in `rq1fct.ipynb` (`DATA_PATH`)
- EOL raw file path variable in `rq1eol.ipynb` (`EOL_PATH`)
- EOL validation raw path variable in `eolval.ipynb` (`VAL_PATH`)

### 5.2 Preprocessed files expected by downstream notebooks
#### EOL chain
Generated by `rq1eol.ipynb` and consumed later:
- `eol_cleaned.csv`
- `eol_outcome_matrix.csv`
- `eol_cost_vector.csv`
- `eol_unit_summary.csv`

#### FCT chain
Consumed by `rq2fct.ipynb` and `rq3fct.ipynb`:
- `fct_cleaned.csv`
- `fct_outcome_matrix.csv`
- `fct_cost_vector.csv`
- `fct_unit_summary.csv`

Validation notebook `valfct.ipynb` also expects:
- `fct_val_cleaned.csv`
- `fct_rq2_cred.csv` (produced by earlier RQ notebooks)

```csv

```

## 8. Practical Setup Tip
Before running notebooks, copy all required input CSV files into the repository root (or update notebook path variables to your local data directories). This avoids path-related failures and improves reproducibility.
